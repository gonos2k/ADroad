# 프레임워크 재검토 — JAX vs PyTorch (동적 연산그래프·모델 크기 관점)

> 질문: "모델 크기를 고려할 때, 동적 연산그래프가 중요하다면 PyTorch가 더 적절한가?"
> 검토일: 2026-07-02 · 대상: dROAD 설계계획서 v0.2

## TL;DR
**이 모델에 한해서는 JAX가 여전히 우세하다.** 이유가 역설적인데, **모델이 작다는 사실 자체가 JAX에 유리**하게 작용하고, RoadSurf의 제어흐름은 "동적 그래프"가 필요한 종류가 **아니기** 때문이다. PyTorch가 실제로 유리해지는 조건은 좁고 명확하다(아래 §4). 개발 편의(eager 디버깅)가 결정적이라면, PyTorch로 가기보다 **JAX+Equinox(+ jit 끄고 개발)** 로 그 이점 대부분을 얻는 편이 낫다.

---

## 1. "동적 연산그래프"가 실제로 필요한 경우 vs RoadSurf
동적 그래프(define-by-run)가 값을 발하는 경우는 **그래프 구조가 입력마다 예측 불가하게 바뀔 때**다 — 트리 구조 신경망, 가변 재귀, 반복 횟수가 입력에 따라 달라지는 RL 롤아웃 등.

RoadSurf의 제어흐름은 그렇지 않다:
- if/else가 **데이터 의존적**이긴 하나, **분기의 집합은 고정·유한·사전 파악 가능**하다("눈>0이면…", "T<동결한계면…"). 이는 "동적 구조"가 아니라 **"고정 구조 + 조건문"**이다.
- 이런 형태는 JAX에서 `jnp.where`(양쪽 계산 후 선택) 또는 `lax.cond`(한쪽 계산)로 **정확히 커버**된다. 새로운 그래프가 매 스텝 생기는 것이 아니다.

**진짜로 "동적"인 곳은 딱 하나 — 커플링의 되감기**다. 반복 횟수가 관측 정합에 따라 달라지고 시간 인덱스 `i`가 되돌아간다. 이건 PyTorch의 eager while-loop가 더 자연스럽다. **그러나 계획서는 이미 커플링을 끄고(방안 A) gradient 동화로 대체하기로 했다.** 즉 프로젝트가 실제로 밟는 경로에는 동적 그래프 수요가 없다.

## 2. "모델 크기"는 오히려 JAX 편 — 두 가지 이유
사용자의 직관과 반대로, 크기 요인은 PyTorch가 아니라 JAX를 가리킨다.

**(a) 작은 상태 + 매우 긴 순차 루프 = 컴파일이 이기는 전형.**
상태는 ~25개 실수(온도 17 + 저장항 5 + 소수)에 불과하고, 시간 루프는 수천 스텝(예: 108 h ÷ 30 s ≈ 12,960 스텝)이다.
- PyTorch **eager**는 스텝마다 수십 개의 **초소형 연산**을 Python 디스패치로 실행 → 디스패치 오버헤드가 연산 자체를 압도한다. "작은 op × 긴 루프"는 eager 프레임워크의 **최악 시나리오**다.
- JAX는 `jit`+`lax.scan`으로 롤아웃 전체를 **하나의 컴파일된 커널로 융합** → 작은 상태에서도 빠르다. 이 조합이 정확히 JAX의 스위트스팟.
- (PyTorch도 `torch.compile`/CUDA graph로 완화 가능하나, 데이터 의존 분기가 많으면 graph break가 잦아 이점이 깎인다.)

**(b) `where` 양쪽 계산 비용이 무시할 수준.**
JAX의 흔한 단점은 `jnp.where`가 양쪽 분기를 다 계산한다는 점이다. 하지만 **상태가 작으므로 양쪽 계산도 사실상 공짜**다. 즉 JAX의 주요 약점이 이 모델에선 사라진다.

## 3. 항목별 비교 (이 프로젝트 기준)
| 항목 | JAX | PyTorch | 유리 |
|---|---|---|---|
| 작은 상태 × 긴 루프 성능 | jit+scan으로 융합 | eager 디스패치 오버헤드 큼 | **JAX** |
| 데이터 의존 분기(고정 집합) | where/cond, 소형이라 저렴 | 네이티브 if(단일 궤적) / where(배치) | 대체로 **JAX**, 단일 궤적 개발은 PyTorch가 읽기 쉬움 |
| 진짜 동적(커플링 되감기) | scan 부적합, while_loop 필요 | eager로 자연스러움 | **PyTorch** (단 방안 A면 무관) |
| 긴 시퀀스 역전파 메모리 | `jax.checkpoint`+scan 깔끔 | `torch.utils.checkpoint` 가능 | 동등, JAX 약간 우위 |
| 다지점·앙상블 배치 | `vmap` 최상 | functorch.vmap 존재하나 덜 매끄러움 | **JAX** |
| 암시적 미분/솔버(BLC 고정점, ODE) | diffrax·optimistix·lineax·jaxopt 성숙 | torchdiffeq 등, 생태계 상대적 약함 | **JAX** |
| float64(stiff 스킴 필수) | CPU 지원(x64 플래그) | CPU 지원 | 동등 |
| eager 디버깅 편의 | jit 끄면 가능, trace는 난해 | 최상(print·pdb) | **PyTorch** |
| 하이브리드 ML 생태계(향후) | flax/equinox 충분 | NN 생태계 더 큼 | PyTorch 약간 |

## 4. PyTorch가 실제로 정답이 되는 조건 (정직하게)
아래 중 하나라도 강하게 해당하면 PyTorch(또는 재검토)가 타당하다.
1. **원본 커플링의 가변길이 되감기를 미분가능하게 충실히 재현**해야 함(방안 B/C를 핵심 요구로 승격). → 동적 제어흐름이 본질.
2. **분기 로직을 반복적으로 실험·디버깅**하는 개발 방식을 최우선하고, 팀이 JAX trace 디버깅에 부담. 
3. 프로젝트가 곧 **대형 PyTorch 신경망과 깊게 결합**되는 하이브리드로 전환(주 목적이 DA/보정이 아니라 ML).

현재 계획(목적=DA+보정, 커플링 방안 A, 다지점 vmap, 암시적 솔버 필요)은 **1·3에 해당하지 않고**, 2는 아래 절충안으로 해소된다.

## 5. 절충안 — "PyTorch스러운 JAX"
개발 편의가 걱정이라면 프레임워크를 바꾸지 말고 다음을 쓴다:
- **Equinox**: PyTorch식 모듈/파라미터 트리 문법을 JAX 위에서 제공(가독성↑).
- **개발 중 `jax.disable_jit()`** 로 eager처럼 한 줄씩 디버깅 → 완성 후 jit.
- **optimistix/lineax**(고정점·선형해), **diffrax**(필요 시 ODE화), **jax.checkpoint**(메모리).
이 조합이면 PyTorch의 실질 이점(가독성·디버깅)을 취하면서 scan·vmap·컴파일·암시적미분을 유지한다.

## 6. 결론 및 계획서 반영
- **권고: JAX 유지.** 모델이 작고 루프가 길며 분기가 "고정 구조"라는 점, 다지점 vmap·암시적 솔버 필요라는 점이 모두 JAX를 가리킨다. "동적 그래프" 수요는 방안 A 채택으로 사라진다.
- **단, 트리거를 명문화**: 위 §4의 조건 1(커플링 충실 재현) 또는 3(대형 하이브리드 ML)이 요구사항으로 확정되면 **PyTorch를 재평가**한다. 이 트리거를 계획서 §11(열린 질문)과 리스크 표에 추가한다.
- **개발 편의는 Equinox+`disable_jit`로 흡수**(§4 도구 스택에 반영). 프레임워크 변경 불필요.
- 참고: 이 판단은 적대적 검토의 M1(BLC는 암시적 고정점으로)·C2(부분 암시 스킴 가능성)와도 일관 — 두 경우 모두 JAX의 암시적미분 생태계가 이점.
